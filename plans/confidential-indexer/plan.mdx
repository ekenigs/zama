# Confidential ERC-7984 Indexer + Read API

**Objective:** A single TypeScript monorepo that, from a fresh clone, runs an Envio indexer watching one ERC-7984 confidential token contract on a local fhEVM stack, persists transfer rows (with explicit `pending_decryption` states), decrypts entitled amounts in a dedicated worker via `@zama-fhe/sdk@alpha`, stores cleartext in Docker Postgres, and exposes a Fastify REST API that looks like a normal ERC-20-style indexer to wallet partners.

**Done means:** `pnpm install && pnpm dev` starts Anvil + deployed contract + partner Postgres + Envio indexer + decrypt worker + API; a scripted confidential transfer produces a decrypted row in the DB and the same value via `GET /v1/addresses/:addr/balance` and transfer history; `DECISIONS.md` argues the trade-offs; one happy-path test and one negative test pass; `pnpm lint && pnpm typecheck && pnpm test && pnpm fallow` are green.

**Non-goals for v1:** Multi-contract indexing, Sepolia deployment, production ops (k8s, metrics stack), wallet SDK, write endpoints, GraphQL partner API, real KMS/HSM key management.

</RichText>

<RichText id="events-indexed">

### On-chain events to index

From **OpenZeppelin `openzeppelin-confidential-contracts`** (IERC7984 / `ERC7984ERC20Wrapper`). The indexer watches **one** deployed wrapper contract:

| Event | Purpose |
| --- | --- |
| `ConfidentialTransfer(from, to, amount)` | All movements including mint/burn semantics |
| `Wrap(to, amount, encryptedAmount)` | Shield / wrap from ERC-20 |
| `UnwrapRequested(receiver, unwrapRequestId, encryptedAmount)` | Unshield step 1 |
| `UnwrapFinalized(receiver, unwrapRequestId, encryptedAmount, cleartextAmount)` | Unshield step 2 â€” cleartext amount on-chain |

`UnwrapFinalized.cleartextAmount` can be stored directly; encrypted handles from other events go through SDK decrypt when entitled.

Event signatures are taken from the OpenZeppelin confidential-contracts ABI; reconcile exact selectors in step 1 before writing `apps/indexer/config.yaml`.

</RichText>

<Callout id="decision-dual-db" tone="decision">

**One Postgres 18 server, two databases.** You do not need two Postgres *instances*. A single `postgres:18` container (started by `docker compose` in `dev.sh`) hosts two logical databases:

| Database | Owner | Connection |
| --- | --- | --- |
| `envio` | Envio HyperIndex (sync cursors, generated entities, Hasura) | `ENVIO_PG_DATABASE=envio` (+ user/password/port) |
| `zama` | Partner store (transfers, balances, cleartext) | `DATABASE_URL=postgresql://â€¦/zama` |

An init script (`docker/postgres/init.sql`) runs `CREATE DATABASE envio` and `CREATE DATABASE zama` on first boot. Envio is pointed at the existing server via `ENVIO_PG_*` env vars so it does not spawn a second Postgres container. Partner cleartext never goes into the `envio` database â€” same isolation as separate instances, less Docker overhead.

</Callout>

<Callout id="decision-worker" tone="decision">

**Decrypt in `apps/worker`, not in Envio handlers.** Event handlers only persist rows with `amount_status: pending_decryption` and enqueue affected addresses for balance refresh. `apps/worker` polls `pending_decryption` rows in batches, calls `packages/decrypt.tryDecrypt`, writes cleartext, refreshes `balances` via on-chain `balanceOf`, and logs `decrypt_attempts`. Network-bound KMS calls never block indexing throughput. ACL delegation backfill is the same loop â€” no separate `onBlock` hook required.

</Callout>

<Callout id="decision-balance-derive" tone="decision">

**Balance from on-chain `balanceOf`, not transfer sums.** The worker calls `token.balanceOf(address)` after decrypting transfer amounts for affected addresses, attempts decrypt on the balance handle, and upserts `balances` with `block_number`. Transfer amounts and balance handles are separate FHE objects. Treat `from == 0x0` as `kind: mint` and `to == 0x0` as `kind: burn`; exclude the zero address from balance queries and transfer history.

</Callout>

<Callout id="decision-decrypt-states" tone="decision">

**Never silently drop undecryptable events.** Every indexed transfer is inserted with `amount_handle` and `amount_status: pending_decryption`. The worker upgrades to `decrypted` + `amount_cleartext` on success. On ACL denial â†’ stay pending (`decrypt_attempts.outcome = acl_denied`). On SDK error or timeout (cap ~5 s per batch item) â†’ stay pending for retry. Crash between insert and worker pass is safe â€” the row stays pending and the next poll picks it up.

</Callout>

<RichText id="decryption-approach">

### Decryption approach (steps 3â€“4)

**Ingest (apps/indexer):** Envio handlers parse events and `INSERT` into partner Postgres via `packages/db`. No SDK imports in handlers â€” handlers stay fast and replay-safe.

**Decrypt (apps/worker):** A dedicated Node process runs a poll loop (e.g. every 2 s, batch LIMIT 50):

1. `SELECT` rows where `amount_status = pending_decryption` (or `balances.balance_status = pending_decryption`).
2. Call `packages/decrypt.tryDecrypt({ handle, contract, from, to, blockNumber })` using indexer EOA (`INDEXER_PRIVATE_KEY`).
3. SDK path (alpha): confirm `@zama-fhe/sdk@alpha` local fhEVM config; verify server-side EIP-712 signing (no browser). Use `userDecrypt` when indexer is transfer party; `delegatedUserDecrypt` when ACL delegation exists.
4. On success: update transfer row + call `balanceOf(from)` / `balanceOf(to)` and decrypt balance handles.
5. Log every attempt in `decrypt_attempts` (1-n per transfer).

**v1 fixture model:** happy-path uses indexer EOA as transfer recipient; second fixture grants ACL delegation to exercise worker backfill on previously pending rows.

</RichText>

<RichText id="testing-strategy">

### Testing strategy (step 6)

**Happy path:** Spin Anvil + deploy + `pnpm dev` (indexer + worker + api). Execute wrap + `ConfidentialTransfer` where indexer EOA is `to`. Wait for worker poll. Assert Postgres row is decrypted and API returns matching cleartext.

**Negative test (chosen):** Transfer between two third-party addresses where indexer has **no** ACL rights. Assert: transfer appears in API with `amount.status === "pending_decryption"`; `pendingDecryptions` increments; row is **not** deleted. Rationale: the brief explicitly forbids silent drops â€” this is the highest-risk regression.

</RichText>

<RichText id="risks">

### Risks
